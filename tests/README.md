# Как устроены тесты

Здесь нет цели проверить каждую строку по три раза. Набор собран вокруг мест, где
ошибка действительно испортит аудит: импорт документов, геометрия, подтверждение
пользователя, расчёты, цены, фотографии и итоговый отчёт.

Коротко: мы проверяем всё, что в скилле работает детерминированно. Ответы Vision,
Mapping и аналитика в тестах уже подготовлены — примерно так же, как обычный код
подменяет ответ внешнего API. Дальше проверяется весь наш контур: примет ли скилл
ответ, откажет ли плохому, правильно ли посчитает результат, сохранит состояние и
соберёт артефакты. Насколько хорошо сама LLM распознаёт план или рассуждает по нему,
эти тесты не измеряют; для этого нужны отдельные live-прогоны на реальных примерах.

## Что где искать

| Область | Файлы | Что считаем важным |
|---|---|---|
| Полный сценарий | `test_mvp_workflow.py` | От импорта до HTML, включая отдельное подтверждение |
| План и geometry | `test_mvp_geometry.py`, `test_mvp_confirmation.py` | Строгая схема, review, ревизии и явные исправления |
| Расчёты | `test_mvp_calculations.py`, `test_doors.py`, `test_mvp_audit.py` | Площади, проёмы, допуск и общие двери |
| Mapping и цены | `test_mvp_mapping.py`, `test_mvp_prices.py` | Построчное сопоставление и независимые проверки цены и количества |
| Фото | `test_mvp_visual.py`, `test_visual.py` | Один Vision на фото и осторожные формулировки без лишних выводов |
| Аналитика | `test_mvp_insights.py`, `test_insights.py` | Гипотезы только по сохранённым фактам |
| Артефакты | `test_mvp_report.py`, `test_mvp_provenance.py` | Читаемый отчёт, экранирование текста и воспроизводимые SHA |
| Контракт с runtime | `test_mvp_registration.py`, `test_mvp_security.py` | Публичные tools, безопасные пути и чистый payload |

Несколько тестов смотрят на внутренние функции напрямую. Это сделано только для
коротких детерминированных правил, которые проще проверить без подготовки полного
job. Основное поведение всё равно проходит через публичные tools.

## Запуск

В опубликованном репозитории:

```bash
python -m pip install -r requirements-test.txt
python -m pytest
```

Проверка покрытия:

```bash
python -m coverage run --source=skill -m pytest
python -m coverage report --show-missing
```

Если каталог тестов запускается отдельно от репозитория, путь к payload задаётся
явно:

```bash
CONSTRUCTION_AUDIT_SKILL_DIR=/path/to/construction_audit_mvp python -m pytest tests/construction_audit_mvp
```
