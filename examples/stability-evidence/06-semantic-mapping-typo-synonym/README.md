# Сценарий 06: опечатка и смысловой синоним в названиях работ

Фактический запуск `audit_10_2_20260719` от 19 июля 2026 года, версия скилла `0.7.7`. В корректную по количествам и ценам смету намеренно внесены два неточных названия работ:

- строка 6: `Монтаж плнтуса` вместо `Монтаж плинтуса`;
- строка 11: `Укладка ламината` вместо поддерживаемой MCP-работы `Устройство пола`.

Ожидаемый результат — не пометить строки как неподдерживаемые и не сравнивать их с неверными работами, а передать неоднозначное сопоставление Mapping-субагенту и затем провести детерминированную проверку по проверенной схеме сопоставления.

**Коротко:** одна работа была написана с опечаткой, а другая — близкими по смыслу словами. Обычного точного сравнения названий оказалось недостаточно, поэтому отдельный этап сопоставления правильно понял обе строки. После этого все 21 позиция прошли проверку, и ложных расхождений не появилось.

## Вход

- [`plan.png`](input/plan.png) — план трёх помещений;
- [`estimate.xlsx`](input/estimate.xlsx) — смета на 21 строку с опечаткой и близким по смыслу названием.

## Как скилл понял неточные названия

Точное детерминированное сравнение названий с canonical works и каталогом MCP не могло разрешить две изменённые строки. Manifest фиксирует реальный `mapping_task_id: 88a8ff35`, а не специальное значение `deterministic_mapping`, и отдельный delegation token. Полученный [`mapping.json`](output/mapping.json) прошёл schema v3 validation.

Субагент сохранил следующие решения:

| Строка | Работа сметы | Canonical work | MCP ID | Confidence | Основание |
|---:|---|---|---|---:|---|
| 6 | `Монтаж плнтуса` | `Монтаж плинтуса` | `baseboard_installation` | 1.0 | Опечатка в слове «плинтуса», единица совпадает |
| 11 | `Укладка ламината` | `Устройство пола` | `floor_installation` | 1.0 | Укладка ламината является видом устройства пола, единица совпадает |

Оба сопоставления видны в [`report.html`](output/report.html) в блоках **Mapping работ для количества** и **Mapping с каталогом MCP**. Опечатка также сохранена как служебное info-сообщение, а не как строительное расхождение.

## Что показывает кейс

LLM используется только для ограниченной смысловой задачи Mapping. После сохранения и валидации соответствий количества, цены и стоимости рассчитываются deterministic-кодом. Корректная смета остаётся корректной: вариативность названий не создаёт ложных findings и не снижает coverage.

При повторном запуске текст rationale или task ID субагента могут отличаться, но ожидаемы те же canonical соответствия и нулевые расчётные расхождения.
