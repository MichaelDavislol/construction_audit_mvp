# Case 05: отсутствующая работа по установке двери

Фактический запуск `audit_10_20260719` от 19 июля 2026 года, версия скилла `0.7.7`. Перед запуском из исходной сметы намеренно удалена строка установки двери для помещения «Кассовая зона». Ожидаемый результат — выявить недостаточное общее количество дверных работ, а после отдельного запроса пользователя сформировать контрольную смету по подтверждённой geometry.

## Вход

- [`plan.png`](input/plan.png) — план трёх помещений с тремя уникальными дверями;
- [`estimate.xlsx`](input/estimate.xlsx) — смета на 20 строк с намеренно отсутствующей работой.

Удаление строки для «Кассовой зоны» является известным устройством теста, а не выводом модели. В опубликованной смете остались две дверные строки: `2` установки для «Торгового зала» и строка с количеством `0` для «Склада».

## Фактический результат

| Метрика | Значение |
|---|---:|
| Geometry revision | 1, подтверждена |
| Помещений / площадь пола | 3 / 210,78 м² |
| Уникальных дверей / окон | 3 / 3 |
| Строк исходной сметы | 20 |
| Coverage количества / цены | 100% / 100% |
| Дверей в смете / на плане | 2 / 3 |
| Findings | 1 high |
| Расчётный финансовый эффект | 7 000 |
| Analyst hypotheses | 2 |
| Строк контрольной сметы | 19 |
| Дверей в контрольной смете | 3 |

Deterministic audit создал `quantity_understatement` высокой важности: по строкам 7 и 20 смета содержит суммарно `2` установки, тогда как подтверждённая geometry содержит `3` уникальные двери. Finding связан с `object_total:door_count` и расчётной трассой строк 7–20.

Машиночитаемая сводка: [`result-summary.json`](result-summary.json). Полный пользовательский результат: [`report.html`](output/report.html). Независимо сформированная контрольная смета: [`generated_estimate.xlsx`](output/generated_estimate.xlsx).

## Корректная граница вывода

Сам аудит намеренно не утверждает, в каком именно помещении пропущена работа: дверные установки проверяются по уникальному количеству на всём объекте, а распределение по помещениям принято из сметы и отдельно не доказывается. Поэтому известный тестовый дефект относится к «Кассовой зоне», но автоматический finding корректно сообщает только доказуемое расхождение `2` против `3`.

Analyst сформулировал более узкую гипотезу о возможном полном пропуске позиции для одного помещения и явно указал ограничение: без проверки у сметчика нельзя установить причину отсутствующей двери.

## Контрольная смета

После отчёта пользователь отдельно попросил сформировать контрольную XLSX. `generate_estimate` использовал подтверждённую geometry и сохранённые цены MCP и создал объектную строку:

| Работа | Количество | Цена | Итого |
|---|---:|---:|---:|
| Установка дверей | 3 шт. | 7 000 | 21 000 |

Таким образом, один запуск показывает обе стороны workflow: обнаружение отсутствующего объёма в исходной смете и создание независимого варианта с правильным количеством уникальных дверей.

В [`output/`](output/) находятся все 15 файлов фактического output-каталога, включая calculation trace, findings, Analyst artifacts, HTML и контрольную XLSX.
