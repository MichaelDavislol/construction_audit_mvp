> **Главное за минуту.** Набор проверяет не только способность найти заложенную ошибку. В нём есть корректные сметы без ложных замечаний, неточные названия работ, недостающие размеры, отсутствующие строки целого помещения, пользовательское подтверждение ошибочно распознанного плана и недоступный сервис цен. Для каждого запуска явно указаны ожидаемое поведение и фактический результат.

Два основных пользовательских сценария показаны пошагово на GitHub Pages. Для каждого из десяти запусков устойчивости здесь доступны отдельная страница, входные документы и полный набор фактических результатов.

Во всех семи завершённых аудитах исходной XLSX проверены все фактически существующие строки — отдельно по объёмам и ценам. Два сценария без исходной сметы сформировали новые расчётные XLSX, а сценарий с недоступным каталогом остановился до начала проверки.

## Что воспроизводится строго, а что может отличаться

| Воспроизводится после подтверждения данных | Может немного отличаться при новом запуске модели |
|---|---|
| Формулы и рассчитанные количества | Распознавание отдельных деталей плана до проверки пользователем |
| Сравнение объёмов, цен и стоимости строк | Формулировки объяснений и модельных наблюдений |
| Проверка всех строк исходной сметы | Текст смыслового сопоставления неточных названий |
| Уникальный учёт дверей и окон | Точный текст технической ошибки внешнего сервиса |
| Безопасная остановка без обязательного каталога цен | Служебные идентификаторы задач и временные метки |
| Полнота сформированной XLSX и список пропущенных позиций | — |

> Vision и смысловые модельные этапы помогают прочитать входные данные, но не являются расчётной границей. Пользователь проверяет геометрию, после чего Python-код выполняет формулы и проверки строго по подтверждённым значениям. Поэтому повторный запуск может немного иначе описать план, но сохранённый расчёт всегда связан с конкретной подтверждённой версией данных.

<details>
<summary><strong>Подробно о воспроизводимости отдельных сценариев</strong></summary>

Сценарии 01 и 03 воспроизводимы при повторном запуске на тех же входных документах: ожидаются соответственно обнаружение заложенных ошибок и проверка без ложных замечаний.

Сценарий 02 практически невозможно повторить побайтово или гарантированно получить одним новым запуском. Модель распознавания изображения случайным образом не связала дверь и проём `Д-4` с подсобным помещением, хотя увидела эту дверь в помещении «Склад». При новом запуске тот же план может быть прочитан иначе. Воспроизводимая часть начинается после подтверждения пользователя: скилл фиксирует принятую геометрию, проводит все расчёты строго по ней и не пытается самостоятельно добавить отсутствующую дверь.

Сценарий 05 повторяемо создаёт расхождение: в смете учтены две установки, а на подтверждённом плане есть три двери. Скилл не приписывает пропуск конкретному помещению без достаточных доказательств, а новая контрольная смета независимо восстанавливает количество `3`.

В сценарии 06 повторяемым результатом является правильное сопоставление опечатки и смыслового синонима, после которого расчёт не находит расхождений. Текст объяснения и служебный номер задачи могут измениться между запусками.

Сценарий 07 повторяется отключением сервиса цен или его учётных данных. Точный текст технической ошибки может различаться, но итог должен быть одинаковым: скилл не получает каталог, не проводит расчёты и не создаёт замечания или отчёт.

Сценарий 08 повторяется на плане со стёртыми подписями, если пользователь снова вводит размеры `9,2 × 4,2 м`. Площадь, периметр, стены и плинтус рассчитываются из введённых данных, а не угадываются моделью.

Сценарий 09 повторяется удалением из XLSX всех строк одного помещения. Распознавание должно сохранить это помещение с плана, проверка исходной сметы — не придумывать отсутствовавшие строки, а отдельно созданная новая смета — вернуть помещение по данным плана.

Сценарий 10 использует тот же план и те же введённые размеры, что сценарий 08, но без исходной XLSX. Повторяемый результат — подтверждённая геометрия и новая смета на 31 строку без пропусков.

<details>
<summary><strong>Основные термины</strong></summary>

- **геометрия** — извлечённые из плана помещения, размеры, двери и окна;
- **версия геометрии** — состояние геометрии после распознавания или изменения пользователем;
- **полнота проверки 100%** — проверены все строки исходной сметы, а не только удобная выборка;
- **расхождение** — найденное отклонение или замечание;
- **Mapping** — сопоставление разных формулировок одной и той же работы;
- **MCP-каталог** — внешний справочник поддерживаемых работ и цен;
- **режим «только план»** — исходной сметы нет, и новая смета строится только по плану;
- **контрольная смета** — новый независимый расчёт по подтверждённому плану и ценам MCP;
- **детерминированный расчёт** — программный расчёт по заданным формулам без свободных догадок языковой модели.

</details>

<details>
<summary><strong>Единая структура и содержимое сценария</strong></summary>

Каждая страница начинается с короткого объяснения: что специально изменили, как должен был повести себя скилл и что получилось. Ниже идут точные показатели и исходные файлы для технической проверки.

```text
NN-case-name/
├── README.md
├── result-summary.json
├── manifest.json
├── input/
│   ├── plan.png
│   └── estimate.xlsx  # только для ветки с исходной сметой
└── output/
    └── <полный набор результатов запуска>
```

- `input/` содержит все фактические входные файлы конкретного запуска;
- `output/` перенесён полностью, без отбора «удачных» артефактов;
- `manifest.json` сохраняет версию, статусы, временные метки, хеши, идентификаторы задач и сведения о происхождении данных;
- `result-summary.json` даёт компактные сопоставимые метрики;
- `README.md` фиксирует ожидаемый и фактический исход, чтобы сценарий нельзя было интерпретировать задним числом.

</details>
