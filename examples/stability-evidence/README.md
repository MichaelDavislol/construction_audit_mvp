# Stability evidence: реальные запуски

Этот каталог собирает десять независимых фактических запусков `construction_audit_mvp` в едином формате «вход → полный output». Сейчас добавлены первые семь кейсов, выполненные 18–19 июля 2026 года версией скилла `0.7.7`.

Цель набора — показать устойчивое поведение на разных исходах: скилл находит заложенные ошибки, детерминированно работает с подтверждённой пользователем геометрией и не создаёт findings для корректной сметы.

Во всех пяти завершённых аудитах с исходной XLSX проверено 100% строк как по количеству, так и по цене. Кейс 04 проверяет отдельную plan-only ветку, а кейс 07 — безопасную остановку до аудита при недоступности обязательного MCP. Набор включает положительные срабатывания, контрольные чистые результаты, semantic fallback, fail-closed поведение и проверку workflow gates; это важнее для демонстрации стабильности, чем серия только «успешных находок».

## Воспроизводимость

Кейсы 01 и 03 воспроизводимы при повторном запуске на тех же входных документах: ожидаются соответственно обнаружение заложенных ошибок и чистый аудит без findings.

Кейс 02 практически невозможно воспроизвести побайтово или гарантировать повторением одного запуска. Ошибка возникла на стохастическом Vision-этапе: модель не связала дверь и дверной проём `Д-4` с подсобным помещением, хотя распознала эту дверь в помещении «Склад». При новом запуске Vision может извлечь геометрию иначе. Воспроизводимая часть этого кейса начинается после пользовательского подтверждения: скилл фиксирует принятую geometry revision, рассчитывает все проверки строго по ней и не пытается самостоятельно дополнить отсутствующую дверь.

Именно поэтому review и отдельное подтверждение геометрии являются обязательной границей workflow: Vision помогает извлечь данные, пользователь отвечает за принятие расчётной базы, а deterministic-часть скилла стабильно следует подтверждённым данным.

В кейсе 05 намеренно удалённая из XLSX работа воспроизводимо даёт объектное расхождение при подтверждённой geometry с тремя уникальными дверями. При этом скилл не приписывает пропуск конкретному помещению без достаточных доказательств, а контрольная смета независимо восстанавливает количество `3`.

В кейсе 06 ожидаемые canonical-соответствия и нулевой deterministic результат воспроизводимы; свободный текст основания и runtime task ID Mapping-субагента могут измениться между запусками.

Кейс 07 воспроизводится отключением MCP или его credentials: конкретный текст инфраструктурной ошибки может различаться, но обязательный результат — отсутствие каталога, Mapping, расчётов, findings и отчёта.

## Единая структура кейса

```text
NN-case-name/
├── README.md
├── result-summary.json
├── manifest.json
├── input/
│   ├── plan.png
│   └── estimate.xlsx  # только для ветки с исходной сметой
└── output/
    └── <полный фактический output job>
```

- `input/` содержит все фактические входные файлы конкретного запуска;
- `output/` перенесён полностью, без отбора «удачных» артефактов;
- `manifest.json` сохраняет версию, статусы, timestamps, hashes, task IDs и runtime provenance;
- `result-summary.json` даёт компактные сопоставимые метрики;
- `README.md` фиксирует ожидаемый и фактический исход, чтобы кейс нельзя было интерпретировать задним числом.

## Санитизация

В этих семи кейсах изменено только поле `documents.plan.vision_source_path` в `manifest.json`: абсолютный локальный путь заменён на `input/plan.png`. Все файлы в `input/` и `output/` перенесены без изменений. Токены, task IDs, timestamps, staged filenames, content hashes и расчётные значения намеренно не редактировались.
