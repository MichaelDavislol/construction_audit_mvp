# Construction Audit MVP

Вести две изолированные ветки:

- XLSX + один план → Vision geometry → полный review → ноль или больше
  детерминированных пользовательских ревизий с новым полным review после каждой
  → новое явное подтверждение пользователя → MCP-каталог цен → Mapping
  → независимые детерминированные Python-проверки объёмов и стоимости
  → обязательное предложение приложить ZIP с 1–5 фото либо явно продолжить без фото
  → отдельный Vision-субагент на каждое фото и visual insights либо зафиксированный пропуск
  → обязательный аналитический LLM-субагент → детерминированная validation гипотез
  → HTML → отдельное необязательное предложение сформировать контрольную XLSX-смету;

- только один план → Vision geometry → полный review → ноль или больше
  детерминированных пользовательских ревизий с новым полным review после каждой
  → новое явное подтверждение пользователя → MCP-каталог цен
  → предложение сформировать предварительную смету
  → отдельное явное согласие пользователя → детерминированная генерация XLSX.

Skill использует A2A-делегацию через Vision- и Mapping-субагентов, строгую schema
validation, фиксацию версии payload по SHA-256, обязательный geometry review и
детерминированный Python-аудит.

Основной агент только оркестрирует workflow. Он не создаёт и не исправляет geometry
или mapping. После `import_documents` либо `import_plan` он не читает attachments или
исходники skill.

## Публичные tools

Использовать ровно четырнадцать tools:

1. `create_case(job_id, object_name)`.
2. `import_documents(job_id, estimate, plan)`.
3. `import_plan(job_id, plan)` — отдельная ветка без исходной XLSX.
4. `save_geometry(job_id, vision_task_id, analysis)`.
5. `confirm_geometry(job_id, geometry_revision, confirmed, corrections?,
   user_statement?)` — подтверждает текущую revision либо детерминированно создаёт
   следующую revision из явных пользовательских исправлений.
6. `render_geometry_review(job_id, geometry_revision)` — после повторного
   handoff-напоминания runtime для Vision-ветки либо сразу после успешной
   пользовательской correction, где субагента и handoff-барьера нет.
7. `save_price_catalog(job_id, catalog_response)` — валидирует object-wrapper либо
   его текстовую runtime-projection и сохраняет каталог из `result`; в audit-ветке
   возвращает детерминированный Mapping для точных совпадений либо Mapping packet,
   в plan-only ветке — предложение генерации.
8. `generate_estimate(job_id)` — после отдельного явного согласия пользователя
   создаёт предварительную XLSX-смету из подтверждённой geometry и цен MCP.
9. `run_audit(job_id, mapping_task_id, mapping, tolerance_percent=5)`.
10. `skip_visual_review(job_id)` — только после отдельного ответа пользователя «без фото».
11. `import_site_photos(job_id, archive)` — импортирует один ZIP с 1–5 фото.
12. `save_visual_analysis(job_id, photo_id, photo_task_id, analysis)` — валидирует
    результат отдельного фото Vision-субагента; последний результат подготавливает
    аналитический LLM packet.
13. `finalize_audit(job_id, insights_task_id, llm_insights)` — валидирует результат
   отдельного аналитического субагента и только после этого создаёт HTML.
14. `render_audit_summary(job_id)` — после любого handoff-напоминания runtime,
   включая напоминание после `finalize_audit` или предыдущего render.

Помимо них в workflow разрешены встроенные `schedule_subagent`, `wait_task` и ровно
один внешний MCP-tool `mcp_construction_prices__get_supported_works`. Делегации
используются для Vision плана, Mapping, фото Vision и аналитического субагентов. Если делегация
или успешное ожидание недоступны, остановить pipeline с `orchestration_unavailable`.
Если MCP-tool отсутствует, недоступен или вернул tool error, остановить pipeline с
`price_catalog_unavailable`: не запускать Mapping, аудит или генерацию сметы. Не
использовать `get_task_result` вместо успешного `wait_task`. Полный `wait_task`
envelope допустим только как явно описанный fallback для `save_visual_analysis`
и `finalize_audit`; в остальные public tools его не передавать.

## Vision

1. Вызвать `create_case`. Если приложены XLSX и план, один раз вызвать
   `import_documents` с обеими целыми staged manifest entries. Если приложен только
   план, один раз вызвать `import_plan` с целой staged manifest entry плана. Не
   создавать пустую или фиктивную XLSX.
2. `vision_delegation` — готовые args для `schedule_subagent`. Передать поля
   `role`, `model_lane`, `memory_mode`, `write_surface`, `objective`,
   `expected_output`, `context` и `constraints` строго 1:1: не пересказывать,
   не сокращать, не объединять и не дополнять их данными исходного сообщения.
   `context` уже является JSON string нужного tool-типу; не разбирать и не
   собирать его заново.
3. Перед вызовом проверить без изменения packet: декодированный
   `context.image_ref` — абсолютный путь внутри `data/uploads/construction_audit_mvp`
   и дословно равен `plan.source_path`. Не использовать относительный `uploads/...`.
   При нарушении остановиться с `orchestration_packet_invalid`, а не выбирать
   другой путь. Первым файловым действием Vision вызывает `view_image` ровно с
   этим путём. Запрещено заменять его исходным attachment path, использовать
   parent `artifact_store` или после ошибки `view_image` искать файл другими tools.
   Дверь трактуется как проём на границе помещений. Если она находится между двумя
   распознанными помещениями, Vision обязан включить её в `doors` обоих помещений с
   одинаковыми `element_id`, шириной и высотой. Направление открывания и положение
   подписи не определяют единственную комнату-владельца. Дверь во внешнее или
   нераспознанное пространство включается только в распознанное помещение. Это
   геометрическая смежность, а не сметная привязка установки. Одинаковый
   `element_id` с разными размерами в комнатах блокирует подтверждение и аудит с
   `shared_door_dimension_conflict`; генерация сметы также запрещена.
4. Сохранить `vision_task_id` и получить успешный результат через `wait_task`.
5. Найти JSON между `[BEGIN_SUBTASK_OUTPUT]` и `[END_SUBTASK_OUTPUT]`, при наличии
   удалить только точный префикс `FINAL ANSWER:` и один внешний набор тех же
   runtime-маркеров. Допустимо снять один внешний Markdown JSON fence. Выполнить
   `json.loads` без исправлений.
6. Передать полученный object в `save_geometry` как `analysis`, а
   `vision_task_id` — только для трассировки.
7. При `geometry_review_required`:
   1. Результат `save_geometry` намеренно не содержит `review_markdown`: review уже
      сохранён, но до handoff-барьера пользователю его не выводить.
   2. Если после завершения текущего Vision-субагента runtime ещё не добавлял
      `[SYSTEM REMINDER]` с `[SUBAGENT_RESULTS]`, не вызывать tools и вернуть ровно
      короткий draft из `assistant_instruction`. Runtime перехватит этот draft и
      добавит результаты субагента; draft пользователю не виден.
   3. После такого reminder вызвать `render_geometry_review` с текущими `job_id` и
      `geometry_revision`. Если reminder текущего Vision-субагента уже присутствует
      в контексте, вызвать renderer сразу, не создавая второй барьер.
   4. Вывести возвращённый `review_markdown` полностью и дословно; не отвечать
      сообщением о состоянии handoff и не утверждать, что review уже показан выше:
      предыдущий assistant draft пользователю не виден.
   5. Не писать подтверждение от имени пользователя.
   6. Не вызывать `confirm_geometry` в текущем task.
   7. Завершить task.
   8. Продолжить workflow только после отдельного нового входящего сообщения
      пользователя с подтверждением.
   Выполнить `next_action` и `assistant_instruction` из результата.

Запрещено вручную создавать, исправлять или дополнять geometry. После первой
`geometry_schema_invalid` выполнить новую Vision-делегацию, передав субагенту все
`details.validation_errors`. Разрешены максимум две Vision-попытки; после второй
ошибки остановиться с `geometry_analysis_failed`. Последний валидный geometry
artifact при ошибке не менять.

Не выводить пользователю raw JSON или fenced JSON результата Vision-субагента.
Он используется только как вход `save_geometry`. Structured `review` не
пересказывать: пользовательский вывод берётся только из `review_markdown`.

## Пользовательские исправления geometry

Сообщение с исправлением не является подтверждением. Если после показанного review
пользователь однозначно задаёт новое значение, первым public-tool действием нового
хода вызвать `confirm_geometry` для показанной текущей revision с `confirmed=false`,
`user_statement`, дословно равным сообщению пользователя, и `corrections`.

Каждая correction содержит ровно `target`, `room_ids`, `element_ids`, `field` и
`value`:

- `target`: `rooms`, `doors` или `windows`;
- `room_ids`: строка `all` либо непустой список `room_NNN`, исходных ID с плана или
  точных однозначных названий помещений из review;
- `element_ids`: строка `all` либо непустой список точных `element_id`; для
  `target=rooms` использовать только `all`;
- для `rooms` допустимы `floor_area_m2`, `length_m`, `width_m`, `perimeter_m` и
  `height_m`; для `doors`/`windows` — только `width_m` и `height_m`;
- `value`: явно заданное пользователем положительное число либо `null`, если
  пользователь явно сообщил, что значение отсутствует.

Например, «Высота всех окон 1,6 метра» переводится только в следующую операцию:
`target=windows`, `room_ids=all`, `element_ids=all`, `field=height_m`, `value=1.6`.
Основной агент формирует только эти ограниченные селекторы; geometry JSON он не
читает, не копирует и не редактирует. Tool сам применяет правку ко всем совпавшим
полям, фиксирует исходное сообщение и нормализованное значение в provenance,
пересчитывает площадь и периметр при наличии длины и ширины, удаляет относящиеся к
исправленным полям устаревшие warnings, обновляет conflicts и создаёт revision `N+1`.

После успешной правки немедленно вызвать `render_geometry_review` с возвращённой
revision, полностью и дословно вывести `review_markdown` и завершить task. Для этой
ветки handoff-барьер и `[SUBAGENT_RESULTS]` не нужны: субагент не запускался. Не
вызывать `confirm_geometry(... confirmed=true)` в том же task.

Последовательные исправления не ограничены номером revision: новая явная правка
после revision 3 создаёт revision 4, затем 5 и так далее. Всегда передавать текущую
показанную revision; stale revision должна завершаться `stale_geometry_revision`.
Оставаться в том же `job_id`: запрещено повторно вызывать `create_case`,
`import_documents`, `import_plan`, `schedule_subagent`, `wait_task` или `save_geometry`, заново
распознавать план, читать attachments/XLSX или искать исходный файл.

Если исправление неоднозначно — например, непонятно, к какому окну относится размер,
— не вызывать tools, запросить уточнение и завершить task. Не угадывать selectors.
Если geometry ранее уже была подтверждена или аудит завершён, correction разрешена:
tool инвалидирует подтверждение, сохранённый MCP-каталог, downstream Mapping/audit/HTML
и generated estimate. После review новой revision нужны новое отдельное подтверждение
и новый MCP-каталог. В audit-ветке затем нужны новая Mapping-делегация и новый аудит;
в plan-only ветке — новое предложение и отдельное согласие на генерацию. Vision и
импорт документов всё равно не повторяются.

## Подтверждение

Основной агент не подтверждает geometry от имени пользователя. Только после нового
однозначного сообщения пользователя вызвать `confirm_geometry` с показанной
revision и `confirmed=true`, без `corrections` и `user_statement`. Это должно быть
первым public-tool действием нового хода. До `confirm_geometry` запрещены
`run_script`, `run_command`, `read_file`, `schedule_subagent`, `run_audit`,
`generate_estimate`, чтение XLSX и самостоятельное построение Mapping. Проверка
отдельного task/message остаётся обязательной. Сообщение, содержащее исправление,
нельзя трактовать как подтверждение,
даже если формулировка также выражает общее согласие с остальной geometry: сначала
создать и показать новую revision, затем дождаться отдельного подтверждения.

## Mapping и аудит

1. Сразу после успешного `confirm_geometry(... confirmed=true)` вызвать без
   аргументов `mcp_construction_prices__get_supported_works`. Не запускать Mapping
   до успешного завершения этого вызова.
2. Нормальный object передать без изменений в
   `save_price_catalog(job_id, catalog_response=response)`; он должен содержать
   массив `result`. Если runtime представил ответ строкой с последовательными JSON
   objects, не разбирать её: передать дословно как
   `catalog_response={"result": [], "runtime_text_projection": response}`. Пустой
   `result` здесь служит только schema-compatible envelope и не считается каталогом. Tool сам
   детерминированно восстановит wrapper и отметит
   `source_transport=runtime_text_projection`.
   `save_price_catalog` детерминированно валидирует для каждой позиции `id`, `name`,
   `unit`, `price`, отклоняет неверную структуру и duplicate `id`, сохраняет
   `price_catalog.json` и фиксирует способ получения. При отсутствии `result`,
   не-array `result` или неверном элементе остановить pipeline с ответом tool; не
   чинить каталог основным агентом и не запускать Mapping.
3. Если `save_price_catalog` вернул `status=estimate_generation_offer`, вывести
   `offer_markdown` полностью и завершить task. Не вызывать `generate_estimate` в
   этом task. Дождаться отдельного нового сообщения пользователя. Только при явном
   согласии вроде «да, сформировать смету» вызвать `generate_estimate(job_id)` и
   показать путь из `estimate_artifact`. Это выбор действия, а не повторное
   подтверждение geometry.
4. Если `save_price_catalog` вернул `mapping_mode=deterministic_exact_match`, сразу
   передать возвращённые `mapping` и `mapping_task_id` в `run_audit`. Иначе
   использовать только packet из `mapping_delegation` успешного результата текущей
   подтверждённой geometry revision и передать его в
   `schedule_subagent` без смысловых изменений. Не строить `objective` или
   `context` вручную, не добавлять geometry summary и не читать XLSX повторно.
   Mapping task, созданный до сохранения текущего MCP-каталога, использовать
   запрещено.

   Mapping выполняется построчно и в двух независимых профилях:

   - `work_matches` / `work_unsupported` / `work_unresolved` сопоставляют работу с
     локальной canonical work только для существующей проверки объёмов;
   - `price_matches` / `price_unsupported` / `price_unresolved` сопоставляют ту же
     строку с MCP по смыслу, действию, объекту работы и единице измерения. Успешный
     элемент возвращает только стабильный `mcp_work_id`, а не цену и не название в
     качестве идентификатора.

   Каждый элемент `estimate_works` содержит `source_row`, `name` и `unit` и должен
   быть покрыт ровно один раз в каждом профиле. Решение по одной строке не
   распространяется на одноимённые работы других строк. Mapping-субагенту запрещено
   возвращать, изменять или вычислять `price`; он отвечает только за смысловое
   сопоставление.
5. Сохранить `mapping_task_id`, выполнить `wait_task`, извлечь JSON по правилам
   Vision: точный префикс `FINAL ANSWER:` и один внешний набор runtime-маркеров не
   являются частью JSON. При необходимости снять один внешний JSON fence и выполнить
   `json.loads` без исправлений.
6. Передать object в `run_audit` как `mapping`, а `mapping_task_id` — только для
   трассировки. Внутренний `delegation_token` должен вернуться из Mapping без
   изменений; пользователю его не показывать.

`run_audit` повторно загружает сохранённый catalog, проверяет его SHA-256, для
каждого `mcp_work_id` заново находит позицию и только из неё берёт `price`. Python
считает `quantity × MCP price`, сравнивает цену за единицу и стоимость строки,
считает абсолютные и процентные отклонения и формирует findings при превышении
`tolerance_percent`. Для quantity finding возможное влияние считается только при
единой доступной MCP-цене как `(estimate_quantity - control_quantity) × mcp_price`.
Для цены влияние считается как `(estimate_price - mcp_price) × estimate_quantity`.
Значения не получают обозначение валюты. Отсутствующие цена, количество или
стоимость дают независимый непроверенный/частично проверенный ценовой статус.
Несовместимая единица отклоняет ценовой mapping. Неподдерживаемая MCP работа остаётся
warning и не отключает геометрическую проверку; неподдерживаемая geometry work не
отключает ценовую.

Для `Установка дверей` room-привязка количества принимается как заявленная в смете
и отдельно по помещениям не проверяется. Python суммирует количество по всем
проверяемым строкам этой работы и сравнивает его только с уникальным количеством
дверей объекта, дедуплицированным по `element_id`. Расхождение создаётся одно на
уровне объекта и ссылается на все охваченные строки. При этом один общий дверной
проём участвует в площади проёмов, чистой площади стен и длине плинтуса каждого
смежного помещения. Пользовательское исправление сметного распределения в этой
MVP-версии не поддерживается.

Перед расчётами `run_audit` проверяет внутреннюю целостность derived geometry. Если
длина и ширина есть, а площадь или периметр не рассчитаны, остановить аудит с
`geometry_derived_state_incomplete`. Это ошибка pipeline, а не неполнота данных
пользователя. Если исходных размеров действительно не хватает, аудит продолжить и
оставить только зависимые строки непроверенными.

Не создавать и не исправлять mapping основным агентом. Не выводить пользователю
raw JSON или fenced JSON Mapping-субагента: использовать его только как вход
`run_audit`. Невалидный JSON завершает
pipeline с `mapping_execution_failed`; schema validation error — с
`mapping_schema_invalid`. При `mapping_review_required` показать unresolved
элементы любого профиля и запросить уточнение; расчёты и отчёт не создавать.

## Фотографии объекта

После успешного `run_audit` HTML и аналитический LLM packet ещё не создаются.
Полностью вывести `visual_review_offer_markdown` и завершить task. Дождаться
отдельного нового сообщения пользователя.

- Если пользователь явно ответил «без фото», вызвать `skip_visual_review(job_id)`.
  Tool сохранит зафиксированный пропуск и вернёт `llm_insights_delegation`.
- Если пользователь приложил ZIP, вызвать `import_site_photos` с целой staged
  manifest entry архива. ZIP должен содержать от 1 до 5 PNG/JPG/JPEG; другие
  файлы не являются фотографиями и игнорируются. Не просить отдельные JPG/PNG.
- `visual_delegations` содержит отдельный `{photo_id, packet}` для каждой фотографии.
  Совокупный ответ с максимум пятью packets ограничен безопасным transport-бюджетом;
  packet нельзя сокращать, дополнять placeholder-токенами или восстанавливать из файлов.
  В одном tool-call round вызвать `schedule_subagent` для каждого `packet` строго
  1:1, чтобы runtime запустил burst параллельно, и сохранить соответствие
  `photo_id -> task_id`. Затем обрабатывать строго по одному: `wait_task` для одного
  `task_id` → проверить envelope результата → немедленно вызвать
  `save_visual_analysis` с соответствующими `photo_id` и точным 8-символьным
  `photo_task_id` → только после успешного save ждать следующий task. Не вызывать
  `wait_tasks`, `recent_tasks`, `cancel_task`, `get_task_result` и не повторять
  `import_site_photos`, даже если runtime рекомендует batch wait.
- Основной агент не открывает ZIP, фотографии или сохранённые visual artifacts и
  не создаёт visual insights самостоятельно.
- Фото Vision использует только одну фотографию и компактный список canonical
  works сметы с `source_rows`. Фотография не привязана к помещению: не определять,
  не угадывать и не упоминать комнату. `source_rows` связывают наблюдение только со
  строками соответствующей работы в смете, а не с местом съёмки. Для каждой работы
  из списка создавать ровно один `estimate_comparison` в исходном порядке с её
  точными `estimate_work` и `source_rows`, даже если работа не видна: грунтовка и
  окраска стен, пол, потолок, плинтус, двери и окна. Не создавать два
  `estimate_comparison` для одной canonical work. Для сравнения допустимы только
  `status=observed`, `not_observed` или `not_assessable`; не использовать `matches`,
  `mismatch` или `quality_concern`.
  `not_observed`
  означает только, что результат работы не виден на показанном участке. Не решать,
  нормально ли это, и не утверждать невыполнение. В `title` и `observation` писать
  «не видно в кадре», а не «не выполнено», «не установлено» или «отсутствует».
  Скрытые работы возвращать как `not_assessable`. Признаки трещин, пятен, щелей,
  перекосов и других проблем качества возвращать максимум одним дополнительным item:
  `category=quality`, `estimate_work=null`, `source_rows=[]`,
  `status=quality_concern`. Все поля примера обязательны; не использовать
  `category=quality_concern`.
- Невалидный ответ одного фото не исправлять основным агентом: повторить только
  соответствующий packet из `details.visual_delegation` новым вызовом
  `schedule_subagent` и передать новый `photo_task_id`. Отклонённый task ID
  повторно использовать запрещено. Не пытаться исправить enum, текст, отсутствующие
  поля или повторно отправить analysis: немедленно выполнить `assistant_instruction`
  ошибки и передать возвращённый packet строго 1:1. Retry packet уже содержит
  точные `validation_errors` предыдущей попытки; не пересказывать и не изменять их.
- В `save_visual_analysis.analysis` предпочтительно передавать точную JSON-строку
  между внутренними runtime-маркерами дословно, без `json.loads`, исправлений и
  повторной сборки object. Для обратной совместимости tool также принимает уже
  сформированный object. Если надёжно выделить внутреннюю строку не удалось, можно
  передать полный неизменённый результат `wait_task`: tool детерминированно отличит
  фактический `FINAL ANSWER` от примера в `task_contract`. Ошибка
  `visual_analysis_transport_invalid` не является ошибкой фото-субагента, не
  отклоняет его task ID и не требует нового запуска. Выполнить возвращённый
  `next_action=reextract_same_subagent_result`: заново извлечь object из уже
  полученного результата того же `wait_task` и повторить save с тем же task ID.
- Ответ фото-субагента валиден только если до `FINAL ANSWER:` и после
  `[END_SUBTASK_OUTPUT]` нет ничего, кроме whitespace. Пояснение, postamble или
  второй JSON вне envelope означают повтор только этого photo packet.
- После сохранения последнего фото `save_visual_analysis` детерминированно создаёт
  `visual_insights.json` и возвращает `llm_insights_delegation`.

Visual insights являются отдельными предварительными наблюдениями и не изменяют
детерминированные findings объёма или стоимости. Все visual insights полностью
показываются отдельным разделом HTML. Итоговый аналитический субагент получает
компактную выборку visual insights, равномерно охватывающую предоставленные фото,
и может ссылаться на `visual_insight_id` и соответствующие `source_rows`. Он не
имеет права выводить по фото состояние конкретного помещения или всех помещений.

При `llm_insights_required` HTML ещё не существует. Выполнить обязательный этап:

Если предыдущий task прервался после подготовки `llm_insights_required` и полного
packet больше нет в текущем tool result, вызвать `render_audit_summary(job_id)`.
При сохранённом неизменном контексте tool повторно вернёт полный
`llm_insights_delegation`; продолжить по тем же правилам. Основной агент не читает
и не восстанавливает packet из файлов самостоятельно.

1. Передать `llm_insights_delegation` в `schedule_subagent` строго 1:1. Не сокращать
   context, не добавлять внешние данные и не создавать гипотезы основным агентом.
   Packet из `skip_visual_review` либо последнего `save_visual_analysis` полный и
   заведомо ограничен transport-бюджетом.
   Не искать или восстанавливать его через `list_files`, `search_code`, `read_file`,
   shell либо сохранённые artifacts. При отсутствии полного packet остановиться с
   `orchestration_packet_invalid`.
2. Сохранить `insights_task_id`, получить успешный результат через `wait_task`.
   Предпочтительно дословно выделить JSON-текст между внутренними маркерами, не
   выполнять `json.loads` и не собирать object заново. Если выделение ненадёжно,
   передать полный неизменённый результат `wait_task`.
3. Передать точную JSON-строку либо полный wait envelope в `finalize_audit` как
   `llm_insights`, а task ID — как
   `insights_task_id`. Tool проверяет schema, последовательные insight ID и каждую
   evidence ref относительно сохранённого context package, отклоняет дубликаты,
   неподтверждённую валюту и утверждения об умысле. Для обратной совместимости уже
   сформированный runtime object также допустим. `llm_insights_transport_invalid` не
   расходует аналитическую попытку и не требует нового субагента. Выполнить
   `next_action=reextract_same_subagent_result` с тем же `insights_task_id`.
4. При `llm_insights_schema_invalid` передать возвращённый retry packet в
   `schedule_subagent` строго 1:1, не изменяя ни одного его поля. Разрешены максимум
   три аналитические попытки; после третьей ошибки остановиться с
   `llm_insights_analysis_failed`. Основной агент не исправляет JSON самостоятельно.
5. `status=no_useful_observations` с пустым `items` является корректным результатом:
   отчёт всё равно создаётся. Наблюдения не добавляются в deterministic `findings`.

Поля `summary`, `title`, `observation`, `hypothesis`, `recommended_check` и
`limitations` являются пользовательским текстом отчёта. Аналитический субагент
обязан писать их простым русским языком без внутренних ID, schema keys и enum;
технические ссылки допустимы только в структурированном `evidence_refs`. Validation
отклоняет технические формулировки и требует повторной делегации.

При `audit_completed` после `finalize_audit` вывести `audit_summary_markdown`
полностью, следуя `next_action` и `assistant_instruction`; не заменять его фразой
«Всё чисто», не добавлять собственные визуальные выводы и не вставлять полный HTML.
Возвращённый markdown уже содержит детерминированную сводку фото-наблюдений и
валидированные аналитические гипотезы. После любого `[SYSTEM REMINDER]` с
`[SUBAGENT_RESULTS]`, независимо от того, пришёл он до финализации, после неё или
после предыдущего render, вызвать `render_audit_summary(job_id)` заново и вывести
только возвращённый `audit_summary_markdown` полностью. Не отвечать сообщением о
состоянии handoff и не утверждать, что итог уже показан выше: предыдущий assistant
draft пользователю не виден.

Отчёт автоматически не открывать. Показывать только путь `report_artifact`; не
вызывать `run_command`, `open` или иной способ открытия файла без явной просьбы
пользователя.

После завершённого audit-ветвления `audit_summary_markdown` предлагает отдельную
предварительную XLSX-смету. Не создавать её автоматически. Если пользователь новым
сообщением явно попросил сформировать смету, вызвать `generate_estimate(job_id)`:
подтверждённая geometry и каталог уже сохранены.

## Генерация предварительной сметы

`generate_estimate` использовать как простой MVP без дополнительного Mapping-
субагента: включать только единственную точную пару canonical work + совместимая
единица из сохранённого MCP-каталога. Работы без точной пары или без достаточной
geometry пропускать и учитывать в `skipped_count`; не угадывать соответствия.

Для каждого помещения создавать строки грунтовки и окраски стен, пола, потолка,
плинтуса и окон, если контрольное количество положительно. Установку дверей создавать
одной строкой «Весь объект» по уникальным `element_id`, чтобы не удваивать общий
межкомнатный проём. Цену брать только из сохранённого MCP item, стоимость оставлять
формулой Excel `Количество × Цена за единицу`.

Создавать один лист `Смета` с колонками `№`, `Помещение`, `Наименование работы`,
`Единица`, `Количество`, `Цена за единицу`, `Стоимость`, `Примечание`, полным
диапазоном Excel-таблицы и стилем примера. Не добавлять обозначение валюты: MCP его
не подтверждает.

При любой correction geometry существующая generated estimate инвалидируется вместе
с downstream-артефактами. После нового review и подтверждения новой revision снова
получить MCP-каталог и отдельно предложить пересчёт.

## Ограничения и формулировки

Для проверки объёмов поддерживаются грунтовка/окраска стен, устройство пола, отделка
потолка, монтаж плинтуса, установка дверей и окон. Для проверки стоимости
поддерживаемые позиции всегда определяются текущим валидированным `result` внешнего
MCP. Неподдерживаемые работы остаются warnings. Отсутствующие размеры не подставлять;
отключать только зависимые метрики. В пользовательской части отчёта показывать
русские названия расчётов, оснований и уровней уверенности; технические ID и raw
формулы оставлять только в сворачиваемом техническом приложении.

Использовать формулировки «контрольное расчётное значение», «предварительное
расхождение», «строка требует проверки специалистом». Не утверждать нарушение,
умысел, завышение подрядчиком или «фактическое значение». Результат является
предварительным автоматизированным аудитом и требует проверки специалистом.
В HTML показывать `calculation_trace` как детерминированные русскоязычные формулы,
сгруппированные по помещениям и отклонениям; исходный JSON сохранять только во
вложенном сворачиваемом техническом блоке.
